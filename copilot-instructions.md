# GitHub Copilot Instructions for deductible-tracker

Purpose
-------
This file orients GitHub Copilot to the repository's layout, common development workflows, testing patterns, and places where assistance is most helpful. Use this as a concise reference for suggestion context and constraints when authoring code, tests, or CI changes.

Project overview
----------------
- Language: Rust backend with a static JS frontend and small Node/npm tooling.
- Main binary: `src/main.rs` and `src/lib.rs`.
- HTTP pipeline and bootstrap: `src/main_sections/*` and `src/http/*`.
- DB layers: `src/db/*` (SQLite + Oracle support in `src/db/oracle` and `src/db/sqlite`).
- Routes and handlers: `src/routes/*` and `src/auth_sections/*`.
- Frontend: `static/`, `js/`, and `css/` (vanilla JS using Dexie for local DB sync).
- Migrations & DB bootstrap: `migrations/` and `scripts/bin/migrate.rs`.
- Dockerized dev/test flows: `docker-compose*.yml` at repo root.

When to help
------------
- Suggest small focused edits: idiomatic Rust refactors, function extraction, and lifetime simplifications.
- Provide tests: unit tests for Rust modules and small integration test suggestions (`tests/` folder).
- Suggest Docker and CI improvements only when minimal and backward-compatible.
- Help with SQL queries in `src/db/*` and migration scripts—prefer minimal diffs and explain side effects.

Development workflows (common commands)
--------------------------------------
- Build & run (local):

  cargo build
  cargo run --bin migrate

- Run tests (unit + integration):

  RUST_ENV=development ./scripts/with-oracle-client.sh cargo test -- --nocapture

- Run JS tests:

  npm test

- Docker dev/test (compose):

  docker-compose -f docker-compose.dev.yml up --build
  docker-compose -f docker-compose.test.yml up --build --abort-on-container-exit --exit-code-from test

Environment & secrets
---------------------
- RUST_ENV: `development` or `test` controls environment selection.
- Oracle client: many integration tests require Oracle client tooling; the repo uses `./scripts/with-oracle-client.sh` to wrap commands.
- Docker: local `oracle-dev` container may be used; tests often wait for the container to be healthy before running.

Code style and conventions
--------------------------
- Rust: prefer idiomatic Rust (explicit error handling, `Result`, small helper functions). Keep functions short and focused.
- Modules: group domain logic under `src/db`, `src/routes`, `src/main_sections`.
- Tests: unit tests live next to modules where useful; integration tests are in `tests/`.

Areas where Copilot should be conservative
----------------------------------------
- Avoid guessing or writing new database migrations without a clear migration plan—the team prefers explicit migration SQL in `migrations/`.
- Do not change Docker Compose or CI config in sweeping ways; propose small, reversible edits.
- For Oracle-specific code, suggest guarded changes and include fallback for SQLite testing.

Common tasks & helpful prompts
-----------------------------
- "Create a unit test for `src/db/receipts.rs` validating X behavior." — produce a small test using existing helpers in `tests/rust_helpers.rs`.
- "Refactor this handler for clarity and add error mapping to HTTP responses." — propose a short patch and suggested status codes.
- "Add debug logging around DB insert in `src/db/donations.rs`" — include minimal and gated logging only in development builds.

Common pitfalls & Oracle tips
---------------------------
- Many integration tests depend on Oracle readiness—prefer using `scripts/with-oracle-client.sh` and wait loops used in CI.
- If a test fails only in CI, compare environment variables and container memory/cpu constraints.

Files to look at first
----------------------
- `src/lib.rs` and `src/main.rs` — application entry.
- `src/db/mod.rs` and `src/db/*` — persistence layer.
- `src/routes/*` — HTTP endpoints and route wiring.
- `scripts/with-oracle-client.sh` — how to run commands requiring Oracle client.
- `docker-compose.dev.yml` and `docker-compose.test.yml` — local/dev/test environment composition.

PR and commit guidance
----------------------
- Small commits, focused diffs, and descriptive messages help reviewers. Suggest tests for behavioral changes.
- When proposing changes to DB schemas, include migration SQL and describe expected upgrade steps.

If you want more
----------------
Ask for targeted help (e.g., "suggest a unit test for `src/db/receipts.rs`" or "refactor `src/routes/receipts.rs` handler").
